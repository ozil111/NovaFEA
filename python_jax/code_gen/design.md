# Finite Element Code Generator - Design Document

## 1. Overview

This project is a software tool for the automated generation of efficient, low-level code (C++, CUDA, JAX) for Finite Element Analysis (FEA). It leverages the SymPy symbolic mathematics library in Python to serve as a "Single Source of Truth" for complex mathematical formulas.

The primary goal is to allow developers to define new elements and materials using high-level mathematical expressions and automatically generate highly optimized computational kernels, minimizing manual implementation errors and accelerating development.

## 2. Core Architecture: Hybrid Decoupling

After evaluating several architectural patterns, the project adopts a **Hybrid Decoupling** model. This architecture was chosen to balance the need for computational performance with the requirement for a scalable and maintainable codebase, avoiding the "combinatorial explosion" of a fully coupled approach.

The architecture separates code generation into two distinct, independent tasks:

-   **Constitutive Kernels (Materials)**: These are functions responsible for calculating the constitutive matrix (D-matrix) from a set of material parameters (e.g., Young's Modulus `E`, Poisson's Ratio `nu`).
-   **Stiffness Kernels (Elements)**: These are functions that perform the core kinematic calculations and stiffness integration (`Ke = ∫ BᵀDB dV`). They take nodal coordinates and a pre-computed D-matrix as inputs.

This design allows `M` materials and `N` elements to be represented by only `M + N` generated functions, rather than `M * N` functions, which is critical for scalability. The most computationally intensive part—the element stiffness integration—remains a single, highly optimized function generated by this tool, thus preserving performance.

## 3. Directory Structure

The project is organized as follows:

-   `sympy_codegen.py`: The main entry point and command-line interface for the code generator. It acts as the "compiler" and dispatcher.
-   `definitions/`: A Python package containing all symbolic mathematical definitions.
    -   `definitions/abc.py`: Defines the Abstract Base Classes (`Element`, `Material`) that enforce a common interface for all definitions.
    -   `definitions/elements/`: Contains concrete element implementations (e.g., `tet4.py`). Each file defines a class inheriting from `Element`.
    -   `definitions/materials/`: Contains concrete material implementations (e.g., `isotropic.py`). Each file defines a class inheriting from `Material`.
-   `static.py`: A JAX-based linear solver. Initially used for development and verification of the coupled kernels. It is not compatible with the current hybrid decoupling architecture but serves as a useful reference for a JAX-based FEA solver.

## 4. Code Generation Workflow

The code generation process is orchestrated by `sympy_codegen.py`:

1.  **Task Dispatch**: The user runs the script with a specific `--task` (`constitutive` or `stiffness`).
2.  **Dynamic Loading**: Based on the task, the script dynamically imports the required `Element` or `Material` class from the `definitions` directory using the `--element` or `--material` name.
3.  **Model Creation**: The corresponding interface method is called:
    -   `get_constitutive_model()` for a material.
    -   `get_stiffness_model()` for an element.
4.  **Symbolic Representation**: The method returns a `MathModel` object, which is a data container for the symbolic inputs, outputs, and metadata (like human-readable input names).
5.  **Compilation**: The `FEACompiler` class takes the `MathModel` and performs the following:
    -   **Optimization**: Applies `sympy.cse` (Common Subexpression Elimination) to the list of output expressions to create an optimized computation graph.
    -   **Code Printing**: Uses a language-specific printer (`sympy.printing.ccode` for C++/CUDA, `JaxPrinter` for JAX) to convert the symbolic expressions into source code.
    -   **Documentation**: Generates a Doxygen-style comment block detailing the function's signature, including the precise layout of the `in` and `out` arrays.
    -   **File Output**: Wraps the generated code in a function and writes it to the target file.

## 5. How to Extend the Tool

The architecture is designed for easy extension.

### Adding a New Element

1.  Create a new file in `definitions/elements/`, for example, `hex8.py`.
2.  Inside, create a class that inherits from `Element`: `class Hex8(Element):`.
3.  Implement the `get_stiffness_model(self)` method.
4.  Inside this method:
    -   Define symbolic inputs for nodal coordinates (e.g., 24 for Hex8) and the D-matrix components (36).
    -   Symbolically define the element's formulation (shape functions, B-matrix, Jacobian, etc.).
    -   Symbolically compute the stiffness matrix: `Ke = ∫ (Bᵀ * D * B) dV`. For higher-order elements, this will involve setting up a numerical quadrature loop symbolically.
    -   Return a `MathModel` object containing all the necessary information.

### Adding a New Material

1.  Create a new file in `definitions/materials/`, for example, `neohookean.py`.
2.  Inside, create a class that inherits from `Material`: `class Neohookean(Material):`.
3.  Implement the `get_constitutive_model(self)` method.
4.  Inside this method:
    -   Define symbolic inputs for the material's parameters.
    -   Symbolically define the computation for the 6x6 constitutive matrix `D`.
    -   Return a `MathModel` object.
